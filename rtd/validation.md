# Validation and tests

ICARION has two complementary test layers:

1. **CTests** in `tests/` for fast regression testing during development.
2. **Validation suite** in `validation/` for physics, instrument, reaction, and performance validation.

Use CTests to check whether a build is internally consistent. Use `validation/` when you want scientific validation outputs such as tables, plots, and benchmark metrics.

---

## CTests: fast regression tests

After building ICARION with CMake, run the C++ test suite from the build directory:

```bash
cd build
ctest --output-on-failure
```

For a parallel run (if OpenMP is available):

```bash
ctest -j 8 --output-on-failure
```

For a specific test pattern:

```bash
ctest -R gas_flow --output-on-failure
ctest -R reaction --output-on-failure
ctest -R hdf5 --output-on-failure
```

Typical CTest use cases:

- check that a new build works,
- detect regressions after changing code,
- test config parsing and HDF5 output,
- test core physics components on small cases,
- run CI/CD checks quickly.

CTests should normally run in seconds to minutes, depending on build options and hardware.

---

## Validation suite: scientific validation

The `validation/` directory contains higher-level scripts and configs for physics validation. These runs are more expensive than CTests and are intended to produce interpretable outputs.

From the repository root:

```bash
cd validation
```

Primary runners:

```bash
# Instrument validations
./scripts/run_instrument_suite.sh
./scripts/run_instrument_analysis.sh

# Physics validations: thermalization, transport, reactions, IPM, TIMS, space charge
./scripts/run_physics_suite.sh
./scripts/run_physics_analysis.sh

# CPU performance validation/benchmarking
./scripts/performance/run_performance_suite.sh
./scripts/performance/run_performance_analysis.sh
```

The `*_suite.sh` scripts run simulations and write raw outputs. The matching
`*_analysis.sh` scripts turn those outputs into summaries, plots, or pass/fail
metrics. Depending on the selected suite and hardware, the full validation suite
can take much longer than CTests.

## What is validated where?

| Area | CTests | `validation/` suite |
|---|---|---|
| Config parsing | yes | indirectly through validation configs |
| HDF5 output structure | yes | yes, through generated outputs |
| RNG/reproducibility | yes | yes, for repeated validation runs |
| IMS drift / mobility | selected regression tests | instrument validation and analysis scripts |
| Quadrupole / linear ion trap / Orbitrap / TOF | selected tests | instrument suite |
| Thermalization / collision models | selected tests | physics suite |
| Reactions | selected tests | reaction validation scripts, including equilibrium-linked reverse reactions |
| InteractionPotentialModel | selected tests | IPM validation with freshly generated mini sample tables |
| TIMS | selected tests | TIMS elution validation |
| Multi-event collisions | selected tests | physics suite |
| Space charge | selected tests | physics/space-charge suite |
| Performance | optional/limited | performance suite |

## Validation directory layout

The validation directory is organized as follows:

```text
validation/
├── configs/
│   ├── instruments/
│   ├── physics/
│   │   ├── thermalization/
│   │   ├── transport/
│   │   ├── spacecharge/
│   │   └── reactions/
│   └── performance/
├── scripts/
│   ├── instrumentation/
│   ├── physics/
│   ├── performance/
│   └── thermalization/
├── results/
├── logs/
├── figures/
└── runs/
```

Important conventions:

- `configs/` contains validation input configurations. Some are committed; others are generated by scripts.
- `scripts/` contains runners, generators, and analysis scripts.
- `results/`, `logs/`, and `figures/` contain generated validation output.
- `runs/<run-id>/` is the recommended structure for clean per-run output.

## Running individual validation areas

### Instruments

```bash
cd validation
./scripts/run_instrument_suite.sh
```

This covers instrument-type validations such as IMS, linear ion traps, Orbitrap, RF quadrupoles (ion guide or mass analyzers), TOF, and FT-ICR test cases where available.

### Physics

```bash
cd validation
./scripts/run_physics_suite.sh
```

The physics runner covers physical behavior such as collisional thermalization, transport, reactions, and space charge.

Run only a focused subset if supported by the runner:

```bash
./scripts/run_physics_suite.sh reactions
./scripts/run_physics_suite.sh ipm
./scripts/run_physics_suite.sh tims
./scripts/run_physics_suite.sh multi_event
./scripts/run_physics_suite.sh spacecharge
```

### Thermalization

```bash
cd validation
python3 scripts/thermalization/generate_thermalization_configs.py
./scripts/thermalization/run_thermalization_tests.sh full
```

Thermalization tests are useful for checking whether stochastic collision models relax ions toward the configured gas temperature.

### Performance

```bash
cd validation
./scripts/performance/run_performance_suite.sh
```

For v1.1, treat CPU performance as the validated baseline. GPU-related scripts may exist, but GPU runtime behavior should be interpreted according to the current release notes and build configuration.

## Per-run output folders

For reproducible validation, prefer per-run folders rather than overwriting global result folders.

Examples:

```bash
cd validation

./scripts/run_physics_suite.sh --run-id my_physics_check
./scripts/run_instrument_suite.sh --run-id my_instrument_check
./scripts/performance/run_performance_suite.sh --run-id my_performance_check
```

This creates output under a run-specific folder such as:

```text
validation/runs/my_physics_check/
├── logs/
├── figures/
└── results/
```

Use a meaningful `run-id` when comparing branches or preparing publication material.

## Suggested workflow before opening a PR

For documentation-only changes:

```bash
mkdocs build --strict
```

For code or config changes:

```bash
cmake --build build -j
cd build
ctest --output-on-failure
```

For changes affecting physics:

```bash
cd validation
./scripts/run_physics_suite.sh --run-id branch_name_physics
./scripts/run_instrument_suite.sh --run-id branch_name_instruments
```

For changes sensitive to performance:

```bash
cd validation
./scripts/performance/run_performance_suite.sh --run-id branch_name_perf
```

## How to interpret failures

### CTest failure

A CTest failure usually means a regression or a local build/configuration problem. First steps:

1. Re-run only the failing test with `ctest -R <name> --output-on-failure`.
2. Check whether required optional dependencies were enabled/disabled consistently.
3. Check whether the test depends on generated files or local paths.
4. If the failure appears after a code change, inspect the smallest related module first.

### Validation failure

A validation failure can mean either:

- a real physics/model regression,
- a changed output format,
- a changed baseline,
- too few particles/statistics,
- a configuration or generated-file mismatch.

Before interpreting a validation failure as a physics problem, check the generated config, HDF5 metadata, git hash, seed, and run logs.

---

## Publishing validation results

Do not commit large generated HDF5 outputs unless there is a specific reason. Prefer:

1. run validation into `validation/runs/<run-id>/`,
2. export or archive a compact result bundle,
3. include plots, CSV summaries, and exact configs,
4. keep large raw files in an external dataset repository if needed.

This keeps the Git repository usable while preserving reproducibility for papers and releases.
